# PSoC C3 Engineering Log

## 2026-04-03

- The VU descriptor format in the PDL matches the package comments: `data0` carries opcode and word counts, `data1`/`data3` are operand pointers, and `data2` is an immediate shift count for `VU_LSR`, not a pointer fetch.
- The first VU rewrite had two separate problems: a write-timing bug on the AHB master side and a deeper architectural mismatch for `LSR`/`MUL`. Replacing it with a buffered fetch/compute/writeback model was simpler and easier to verify than patching the partial streaming FSM.
- A small standalone `vu_engine_tb` with an in-test AHB memory model was enough to prove the bugs and validate the replacement. The current regression covers `MOV`, `ADD` with carry propagation, `LSR` immediate across word boundaries, and `MUL` with non-zero destination memory.
- `aes_engine.sv` did not need a package-wide import of `aes_accel_pkg`; leaving it in created ambiguous AHB constant references once `cryptolite_pkg` was introduced.
- `sha_engine.sv` had the same `automatic logic` block style that `iverilog` rejects inside procedural blocks. Converting the schedule and round temporaries into explicit combinational signals made the file compile cleanly with the rest of `psoc_c3`.
- The broader `psoc_c3` simulation scaffolding is still incomplete. `psoc_c3/sim/Makefile` referenced `../tb/cryptolite_tb.sv`, but no top-level `psoc_c3` testbench exists yet. A focused `make unit-vu` target now exists so the VU path has a working reproducible check.
- Added `psoc_c3/tb/cryptolite_tb.sv` as a directed top-level harness that parses `+UVM_TESTNAME=...` and covers the regression names already listed in `psoc_c3/sim/Makefile`. It uses a simple AHB-Lite CPU driver plus an in-test AHB master memory model rather than a fake UVM shell.
- The CPU AHB driver was initially wrong in a subtle way: it held `HTRANS=NONSEQ` for both address and data phases, which caused the regfile to capture each MMIO transaction twice. Driving one address phase and then returning to `IDLE` during the data phase fixed both front-door readback and operation-start behavior.
- The TRNG AP monitor had a real bug under simulation: the `ap_cnt_q == ap_window - 1` formulation never terminated the programmed window in the current model. Replacing that with explicit `ap_window_valid` / `ap_window_last` wires made `cryptolite_trng_ap_test` pass and kept the AP configuration readable from MMIO.
- Running all directed tests in one sim exposed state contamination across cases even though the Makefile runs one test per process. Adding a `reset_env` step at the start of each directed test keeps the standalone `vvp` full-suite path honest.
- Replaced the stopgap directed `cryptolite_tb` with a real UVM top. The `psoc_c3` bench now reuses the existing `psoc6` AHB and IRQ agents, adds a dedicated `cryptolite_mem_if` for the DUT master port, and binds everything through `uvm_config_db` instead of test-specific procedural glue.
- The new `psoc_c3` UVM layout is intentionally thin: active AHB agent, passive IRQ agent, memory-model interface, environment, virtual sequencer, basic scoreboard, basic front-door coverage, feature sequences, and one named test per Makefile regression target. That was enough to get native-scheduler Cadence coverage without inventing a fake register model or monitor stack.
- `xrun` turned out to be stricter than `iverilog` about default timescales. The correct fix for this bench was to set a simulator-level default timescale for elaborate/run, not to spray ``timescale`` directives through every imported RTL and VIP file.
- Verified the full `psoc_c3` UVM regression with Xcelium (`make regress SIM=xcelium`). All named tests pass: AES ECB, SHA schedule/process/full, TRNG basic/VN/RC/AP, VU add/mul, interrupt, and busy-blocking.
